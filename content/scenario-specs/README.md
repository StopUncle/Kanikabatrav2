# Scenario specs

Markdown files in this directory describe scenarios to be generated by
the LLM scaffold (`scripts/generate-scenario.ts`). Each file is one
scenario.

## Workflow

1. Copy `example.md` to a new file named after the scenario id.
2. Fill in the frontmatter and the editorial body. Both are read by the
   generator.
3. Run the generator:

   ```bash
   ANTHROPIC_API_KEY=sk-ant-... tsx scripts/generate-scenario.ts content/scenario-specs/<your-spec>.md
   ```

   It writes `lib/simulator/scenarios/<track>/<id>.ts`, lints the file
   structurally in process, then runs `npm run type-check`. Slow step
   (typecheck) can be skipped with `--skip-typecheck` when iterating on
   prose.

4. Review the prose. Wire the new scenario into the track's `index.ts`
   (the generator prints the exact two lines to add). Run
   `npx tsx scripts/validate-scenarios.ts` once more to catch
   duplicate-id collisions across tracks.

5. Commit when satisfied. The generator never commits for you.

## Frontmatter fields

| Field | Required | Notes |
|-------|----------|-------|
| `id` | yes | Lowercase, hyphenated. Becomes the filename and scenario id. |
| `track` | yes | One of `female`, `male-business`, `male-dating`, `anxiety`, `toxic-narc`, `pc-child`, `cluster-b-lab`, `divorce-arc`, `loving-mira`. |
| `level` | yes | 1-based level within the track. |
| `order` | yes | 1-based position inside the level. |
| `title` | yes | Display title. |
| `tagline` | yes | One-line hook. |
| `difficulty` | yes | `beginner`, `intermediate`, or `advanced`. |
| `category` | yes | One of the `ScenarioCategory` values (see `lib/simulator/types.ts`). |
| `estimatedMinutes` | yes | Minutes a typical player takes to finish one run. |
| `tier` | no | `free`, `premium`, or `vip`. Defaults to `free`. |
| `characters` | yes | Array of cast members. Each entry needs `id`, `name`, `description`, `traits`, `defaultEmotion`. |
| `setting` | yes | One paragraph framing the location, time of day, who is present. |
| `opening-beat` | yes | One sentence describing what happens in the first scene before the player makes a choice. |
| `desired-endings` | yes | Array of two to six ending outlines. Each entry is `{outcome: summary}` where outcome is `good`, `neutral`, `bad`, `passed`, or `failed`. |
| `tactic-spotlight` | no | One or more tactic names the scenario teaches. Appears in the engine as `tacticsLearned`. |
| `red-flags-taught` | no | Red-flag slugs the scenario surfaces. |
| `ending-recap-pointer` | no | One line that anchors the editorial intent of the recap. Optional. |

## Body sections (conventions, not enforced)

- **Brief.** Two to three paragraphs setting up the situation, the
  stakes, the asymmetry. The generator uses this to plant the scenario
  in a concrete world.
- **Tactical anchors.** Numbered list of the moves the scenario
  teaches. Each entry is one bold name plus two or three sentences.
  This is the editorial spine.
- **Voice notes.** Tone, register, signature tells of each character.
  Kanika's voice is the through-line; named characters speak in their
  own register.

## Rules the generator enforces

Hard rules baked into the prompt and the structural validator:

- No em-dashes anywhere (project rule, applies to all content).
- Every scene id unique within the scenario.
- Every `nextSceneId` resolves to a real scene.
- No self-loops.
- Ending scenes have `isEnding`, `outcomeType`, `endingTitle`,
  `endingSummary`. They do not have `choices` or `nextSceneId`.
- Every non-ending scene has either `choices[]` or `nextSceneId`.
- Every dialog `speakerId` is `inner-voice`, null, or one of the
  characters declared in the spec.
- Aim for eight to fourteen scenes, two to four endings.

If any of these fail, the generator reports the failures and exits
non-zero. Fix the spec or the prose by hand and re-run.

## Out of scope (for the v1 scaffold)

- Auto-retry on validation failure with the failures fed back into the
  prompt.
- Auto-wiring the track's `index.ts`.
- Daily cron that generates a scenario from a spec queue.
- Cost monitoring across runs.

These are valuable extensions, not blockers. Ship the manual loop first;
once Kanika has written and reviewed a few specs end-to-end, the next
iteration can layer automation on top.
